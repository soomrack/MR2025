Инерциальная навигация~--- один из фундаментальных методов определения
положения и ориентации объекта в пространстве без использования внешних
сигналов~\cite{woodman2007}. В основе метода лежат данные инерциального
измерительного блока (IMU --- \textit{Inertial Measurement Unit}),
включающего трёхосевые акселерометр и гироскоп. Интегрируя угловые
скорости и линейные ускорения во времени, можно восстановить полную
навигационную картину: ориентацию, скорость и положение
объекта~\cite{titterton2004}.

Полётный контроллер Pixhawk является промышленным стандартом в области
автономных летательных аппаратов~\cite{px4_docs}. Он оснащён
высококачественным МЭМС-IMU и поддерживает протокол
MAVLink~\cite{mavlink_proto}~--- открытый телеметрический стандарт,
позволяющий в реальном времени получать сырые данные датчиков на внешний
компьютер~\cite{mavlink_docs}.

Целью данной работы является разработка программной системы, которая
осуществляет чтение сырых IMU-данных (сообщения \texttt{RAW\_IMU}
протокола MAVLink) с полётного контроллера Pixhawk и выполняет их
обработку в реальном времени: вычисление ориентации объекта через
кватернионное интегрирование гироскопа~\cite{kuipers1999}, а также
оценку скорости и позиции посредством двойного интегрирования
ускорения~\cite{groves2013}. Результаты визуализируются в интерактивном
3D-окне с использованием OpenGL~3.3 и библиотеки GLFW~\cite{glfw_docs}.

\textbf{Задачи работы:}
\begin{enumerate}[leftmargin=2cm]
    \item реализовать модуль приёма данных по протоколу MAVLink через
    последовательный порт;
    \item реализовать алгоритм калибровки и кватернионного интегрирования
    гироскопических данных;
    \item реализовать алгоритм двойного интегрирования акселерометра для
    оценки позиции;
    \item реализовать механизм ZUPT (\textit{Zero Velocity Update}) для
    частичной компенсации дрейфа;
    \item создать OpenGL-визуализацию навигационного состояния в реальном
    времени с 3D-моделью, траекторией движения и HUD-панелью графиков;
    \item провести анализ точности полученных результатов.
\end{enumerate}

\subsection{Протокол MAVLink и сообщение RAW\_IMU}

Первым этапом работы стало освоение взаимодействия с полётным
контроллером через протокол MAVLink~\cite{mavlink_proto}.
MAVLink (\textit{Micro Air Vehicle Link})~--- лёгкий бинарный протокол
сериализации для беспилотных систем, разработанный Лоренцем Майером
в 2009 году~\cite{mavlink_docs}. Протокол широко применяется в
экосистеме PX4 и ArduPilot~\cite{px4_docs}.

Для получения сырых показаний датчиков используется сообщение
\texttt{RAW\_IMU} (идентификатор~27 в диалекте \texttt{common}).
Оно содержит девять целочисленных полей в единицах внутренней шкалы
датчика (LSB --- \textit{Least Significant Bit}):

    \label{tab:rawimu}

        >{\centering\arraybackslash}p{3cm}|
        >{\centering\arraybackslash}X|}
        \hline
        \textbf{Поле} & \textbf{Датчик} & \textbf{Описание} \\
        \hline
        \texttt{xacc, yacc, zacc} & Акселерометр & Ускорение по осям X, Y, Z (LSB) \\
        \hline
        \texttt{xgyro, ygyro, zgyro} & Гироскоп & Угловая скорость по осям (LSB) \\

\subsection{Реализация первичного модуля чтения}

В качестве первого шага был написан автономный модуль считывания,
позволявший проверить корректность соединения с Pixhawk и получить
сырые данные для анализа и подбора калибровочных коэффициентов.

Взаимодействие реализовано через последовательный порт
\texttt{/dev/ttyACM0} со скоростью 115\,200~бод.
Инициализация выполнена через POSIX-интерфейс
\texttt{termios}~\cite{posix_termios}: 8 бит данных, без паритета,
один стоп-бит, аппаратное управление потоком отключено, таймаут 1~с.

\begin{lstlisting}[caption={Настройка последовательного порта через termios}]
int setup_serial_port(const char* port, int baudrate) {
    int fd = open(port, O_RDWR | O_NOCTTY);
    struct termios tty;
    memset(&tty, 0, sizeof(tty));
    tcgetattr(fd, &tty);
    cfsetospeed(&tty, baudrate);
    cfsetispeed(&tty, baudrate);
    tty.c_cflag |= (CLOCAL | CREAD);
    tty.c_cflag &= ~CSIZE;
    tty.c_cflag |= CS8;       // 8 data bits
    tty.c_cflag &= ~PARENB;   // no parity
    tty.c_cflag &= ~CSTOPB;   // 1 stop bit
    tty.c_cflag &= ~CRTSCTS;  // no hardware flow control
    tty.c_lflag &= ~(ICANON | ECHO | ECHOE | ISIG);
    tty.c_iflag &= ~(IXON | IXOFF | IXANY);
    tty.c_oflag &= ~OPOST;
    tty.c_cc[VMIN]  = 0;
    tty.c_cc[VTIME] = 10;     // 1 second timeout
    tcsetattr(fd, TCSANOW, &tty);
    return fd;
}
\end{lstlisting}

MAVLink-парсер реализован в виде конечного автомата: функция
\texttt{mavlink\_parse\_char()} возвращает ненулевое значение только
при получении полного валидного пакета с проверкой CRC-16~\cite{mavlink_proto}.

\begin{lstlisting}[caption={Цикл чтения и парсинга MAVLink-пакетов}]
mavlink_message_t msg;
mavlink_status_t  status;
uint8_t buffer[2048];

while (running) {
    int n = read(serial_fd, buffer, sizeof(buffer));
    if (n > 0) {
        for (int i = 0; i < n; i++) {
            if (mavlink_parse_char(MAVLINK_COMM_0,
                                   buffer[i], &msg, &status)) {
                if (msg.msgid == MAVLINK_MSG_ID_RAW_IMU) {
                    mavlink_raw_imu_t imu;
                    mavlink_msg_raw_imu_decode(&msg, &imu);
                    handle_message(&imu);
                }
            }
        }
    }
}
\end{lstlisting}

\subsection{Запись данных в CSV}

Каждый принятый пакет записывается в CSV-файл с меткой времени
высокого разрешения, получаемой через \texttt{gettimeofday()} с
точностью до микросекунды. Принудительный сброс буфера (\texttt{flush()})
выполняется каждые 50~пакетов. Формат строки:

\begin{center}
\texttt{timestamp, AX\_raw, AY\_raw, AZ\_raw, GX\_raw, GY\_raw,
GZ\_raw, MX\_raw, MY\_raw, MZ\_raw}
\end{center}

Данные CSV были использованы для экспериментального подбора смещений
нуля (bias) датчиков по поведению сигналов в состоянии покоя.

\subsection{Результаты первичного чтения}

Сообщения \texttt{RAW\_IMU} стабильно поступали с частотой
порядка 100~Гц, что соответствует штатной конфигурации IMU
на Pixhawk~\cite{px4_docs}. Сырые данные акселерометра в покое
показали устойчивое значение по оси~Z (проекция силы тяжести),
а гироскоп~--- дрейф нуля, подлежащий компенсации на следующем этапе.

\subsection{Обзор проекта}

На основе опыта, полученного при реализации модуля чтения, была
разработана полноценная система \textbf{Pixhawk IMU Visualizer}~---
многопоточное приложение на C++14, в котором два параллельных потока
взаимодействуют через потокобезопасный разделяемый буфер.

\textit{Главный поток} реализует чтение данных: устанавливает
соединение с Pixhawk, непрерывно парсит входящий поток байтов и при
получении пакета \texttt{RAW\_IMU} помещает данные в очередь, а также
записывает их в CSV-файл.

\textit{Поток визуализатора} выполняет навигационные вычисления
и рендеринг: извлекает пакеты из очереди, запускает алгоритм
интегрирования и обновляет трёхмерную сцену на каждом кадре.

\subsection{Разделяемые структуры данных}

Все общие данные инкапсулированы в трёх структурах, описанных в
файле \texttt{imu\_shared.h}.

Структура \texttt{RawIMU} хранит одно необработанное измерение
с меткой времени и шестью целочисленными каналами датчиков:

\begin{lstlisting}[caption={Структура сырого пакета IMU}]
struct RawIMU {
    double  timestamp;
    int16_t xacc, yacc, zacc;    // акселерометр (LSB)
    int16_t xgyro, ygyro, zgyro; // гироскоп (LSB)
    int16_t xmag, ymag, zmag;    // магнитометр (LSB)
};
\end{lstlisting}

Структура \texttt{NavState} хранит полное навигационное состояние:

\begin{lstlisting}[caption={Структура навигационного состояния}]
struct NavState {
    double qw=1, qx=0, qy=0, qz=0; // кватернион ориентации
    double wx=0, wy=0, wz=0;        // угловые скорости (рад/с)
    double vx=0, vy=0, vz=0;        // скорость (м/с)
    double px=0, py=0, pz=0;        // позиция (м)
    double roll=0, pitch=0, yaw=0;  // углы Эйлера (градусы)
    double ax=0, ay=0, az=0;        // ускорение (м/с^2)
};
\end{lstlisting}

Структура \texttt{SharedBuffer} объединяет всё межпоточное взаимодействие.
Каждое поле защищено отдельным мьютексом, что позволяет потокам работать
параллельно без глобальной блокировки всей структуры. Флаг
\texttt{running} объявлен как \texttt{std::atomic<bool>}, поэтому его
чтение и запись потокобезопасны без явного мьютекса:

\begin{lstlisting}[caption={Потокобезопасный разделяемый буфер}]
struct SharedBuffer {

    void add_trail_point(float x, float y, float z) { ... }

\subsection{Калибровка датчиков}

Сырые показания МЭМС-датчиков содержат постоянное смещение нуля (bias),
обусловленное конструктивными особенностями кристалла~\cite{titterton2004}.
Смещения определялись экспериментально по показаниям датчика в покое.
Перевод в физические единицы осуществляется линейным масштабированием:
\begin{equation}
    a_i = \bigl(\tilde{a}_i - b_{a,i}\bigr) \cdot S_a, \quad
    \omega_i = \bigl(\tilde{\omega}_i - b_{\omega,i}\bigr) \cdot S_\omega,
    \label{eq:calib}
\end{equation}
где $\tilde{a}_i$, $\tilde{\omega}_i$~--- сырые целочисленные отсчёты;
$b_{a,i}$, $b_{\omega,i}$~--- смещения нуля; $S_a = 9{,}81 / 3050$
[м/с$^2$/отсчёт], $S_\omega = 0{,}001$ [рад/с/отсчёт].

Числовые значения смещений, определённые экспериментально из
CSV-данных в статике:
\begin{equation}
    \mathbf{b}_a = (31,\; 24,\; 0)^\top, \quad
    \mathbf{b}_\omega = (-7,\; 1{,}5,\; 3)^\top.
    \label{eq:biases}
\end{equation}

Константы вынесены в пространство имён \texttt{Calib} в файле
\texttt{imu\_shared.h}, что позволяет изменять их в одном месте.

\subsection{Представление ориентации кватернионами}

Ориентация тела в пространстве описывается единичным
кватернионом~\cite{kuipers1999}:
\begin{equation}
    \mathbf{q} = q_w + q_x\,\mathbf{i} + q_y\,\mathbf{j} + q_z\,\mathbf{k},
    \quad \|\mathbf{q}\| = 1.
    \label{eq:quat}
\end{equation}

Кватернионы предпочтительнее матриц вращения и углов Эйлера: они не
страдают от шарнирного замка (\textit{gimbal lock}), компактны
(4 параметра против 9) и численно стабильны при нормализации~\cite{kuipers1999}.

\subsection{Интегрирование кватерниона}

Кинематическое уравнение связывает производную кватерниона с угловой
скоростью $\bm{\omega} = (\omega_x, \omega_y, \omega_z)^\top$ в
системе координат тела~\cite{mahony2008}:
\begin{equation}
    \dot{\mathbf{q}} = \frac{1}{2}\,\mathbf{q} \otimes
    \begin{pmatrix} 0 \\ \bm{\omega} \end{pmatrix}.
    \label{eq:qdot}
\end{equation}

Для численного интегрирования на шаге $\Delta t$ применяется точная
формула через ось-угол~\cite{titterton2004}:
\begin{equation}
    \theta = \|\bm{\omega}\| \cdot \Delta t, \qquad
    \Delta\mathbf{q} = \left(\cos\frac{\theta}{2},\;
    \frac{\bm{\omega}}{\|\bm{\omega}\|}\sin\frac{\theta}{2}\right).
    \label{eq:dq}
\end{equation}

Обновление кватерниона и нормализация:
\begin{equation}
    \mathbf{q} \leftarrow \frac{\mathbf{q} \otimes \Delta\mathbf{q}}
    {\|\mathbf{q} \otimes \Delta\mathbf{q}\|}.
    \label{eq:qupdate}
\end{equation}

\subsection{Матрица поворота из кватерниона}

Для перевода векторов из системы координат тела в мировую используется
матрица вращения $\mathbf{R} \in SO(3)$~\cite{kuipers1999}:
\begin{equation}
    \mathbf{R} =
    \begin{pmatrix}
        1 - 2(q_y^2 + q_z^2) & 2(q_x q_y - q_w q_z) & 2(q_x q_z + q_w q_y) \\
        2(q_x q_y + q_w q_z) & 1 - 2(q_x^2 + q_z^2) & 2(q_y q_z - q_w q_x) \\
        2(q_x q_z - q_w q_y) & 2(q_y q_z + q_w q_x) & 1 - 2(q_x^2 + q_y^2)
    \end{pmatrix}.
    \label{eq:rot}
\end{equation}

\subsection{Углы Эйлера (конвенция ZYX)}

Из кватерниона вычисляются углы крена $\phi$, тангажа $\theta$ и
рыскания $\psi$~\cite{titterton2004}:
\begin{align}
    \phi   &= \operatorname{atan2}\!\bigl(2(q_w q_x + q_y q_z),\;
               1 - 2(q_x^2 + q_y^2)\bigr), \label{eq:roll} \\
    \theta &= \arcsin\!\bigl(2(q_w q_y - q_z q_x)\bigr), \label{eq:pitch} \\
    \psi   &= \operatorname{atan2}\!\bigl(2(q_w q_z + q_x q_y),\;
               1 - 2(q_y^2 + q_z^2)\bigr). \label{eq:yaw}
\end{align}

\subsection{Двойное интегрирование (скорость и позиция)}

Вектор ускорения переводится из системы координат тела в мировую
и из него вычитается гравитация~\cite{groves2013}:
\begin{equation}
    \mathbf{a}_\text{world} = \mathbf{R}\,\mathbf{a}_\text{body} -
    \mathbf{g}_\text{world}, \quad
    \mathbf{g}_\text{world} = (0,\; 0,\; -9{,}81)^\top \text{ [м/с}^2\text{]}.
    \label{eq:aworld}
\end{equation}

Последовательным интегрированием методом Эйлера первого порядка
вычисляются скорость и позиция~\cite{woodman2007}:
\begin{align}
    \mathbf{v}(t + \Delta t) &= \mathbf{v}(t) +
    \mathbf{a}_\text{world}(t) \cdot \Delta t, \label{eq:vel} \\
    \mathbf{p}(t + \Delta t) &= \mathbf{p}(t) +
    \mathbf{v}(t) \cdot \Delta t. \label{eq:pos}
\end{align}

\subsection{Метод ZUPT}

Метод ZUPT (\textit{Zero Velocity Update})~--- классический способ
частичной коррекции INS в моменты покоя платформы~\cite{skog2010}.
Если норма угловой скорости опускается ниже порогового значения,
скорость мягко гасится:
\begin{equation}
    \text{если } \|\bm{\omega}\| < \omega_\text{th}:
    \quad \mathbf{v} \leftarrow \mathbf{v} \cdot \alpha_\text{damp},
    \label{eq:zupt}
\end{equation}
где $\omega_\text{th} = 0{,}015$ рад/с~--- порог обнаружения покоя,
$\alpha_\text{damp} = 0{,}85$~--- коэффициент затухания. Дополнительно
при норме мирового ускорения ниже~$0{,}3$~м/с$^2$ ускорение
обнуляется полностью, что устраняет интегрирование шумового сигнала
акселерометра в статике.

\subsection{Функция интегрирования IMU}

Навигационный алгоритм сосредоточен в функции
\texttt{integrate\_imu()} файла \texttt{imu\_visualizer.cpp}.
Она вызывается из потока визуализации для каждого вычитанного пакета.

Первый блок выполняет калибровку сырых отсчётов в соответствии
с формулой~\eqref{eq:calib}:

\begin{lstlisting}[caption={Калибровка сырых показаний}]
double ax = (raw.xacc  - Calib::AX_BIAS)  * Calib::ACCEL_SCALE;
double ay = (raw.yacc  - Calib::AY_BIAS)  * Calib::ACCEL_SCALE;
double az = (raw.zacc  - Calib::AZ_BIAS)  * Calib::ACCEL_SCALE;
double gx = (raw.xgyro - Calib::GX_BIAS)  * Calib::GYRO_SCALE;
double gy = (raw.ygyro - Calib::GY_BIAS)  * Calib::GYRO_SCALE;
double gz = (raw.zgyro - Calib::GZ_BIAS)  * Calib::GYRO_SCALE;
\end{lstlisting}

Второй блок реализует кватернионное интегрирование по формулам
\eqref{eq:dq}--\eqref{eq:qupdate}:

\begin{lstlisting}[caption={Кватернионное интегрирование гироскопа (ось-угол)}]
double theta = std::sqrt(gx*gx + gy*gy + gz*gz) * dt;
if (theta > 1e-10) {
    double norm_w = std::sqrt(gx*gx + gy*gy + gz*gz);
    double s  = std::sin(theta * 0.5) / norm_w;
    double c  = std::cos(theta * 0.5);
    double dqw = c, dqx = gx*s, dqy = gy*s, dqz = gz*s;

    // q_new = q (x) dq
    double qw = n.qw*dqw - n.qx*dqx - n.qy*dqy - n.qz*dqz;
    double qx = n.qw*dqx + n.qx*dqw + n.qy*dqz - n.qz*dqy;
    double qy = n.qw*dqy - n.qx*dqz + n.qy*dqw + n.qz*dqx;
    double qz = n.qw*dqz + n.qx*dqy - n.qy*dqx + n.qz*dqw;

    double norm = std::sqrt(qw*qw + qx*qx + qy*qy + qz*qz);
    n.qw = qw/norm; n.qx = qx/norm;
    n.qy = qy/norm; n.qz = qz/norm;
}
\end{lstlisting}

Третий блок вычисляет матрицу поворота по формуле~\eqref{eq:rot},
переводит ускорение в мировую систему координат, вычитает гравитацию
и применяет ZUPT:

\begin{lstlisting}[caption={Мировое ускорение и ZUPT}]
// Матрица поворота из кватерниона (строки r00..r22)
double ax_w = r00*ax + r01*ay + r02*az;
double ay_w = r10*ax + r11*ay + r12*az;
double az_w = r20*ax + r21*ay + r22*az;
az_w -= (-9.81);   // вычитаем гравитацию

// ZUPT
double gyro_norm = std::sqrt(gx*gx + gy*gy + gz*gz);
if (gyro_norm < Calib::ZUPT_GYRO_THRESH) {
    n.vx *= 0.85; n.vy *= 0.85; n.vz *= 0.85;
    if (std::abs(ax_w) < 0.3) ax_w = 0;
    if (std::abs(ay_w) < 0.3) ay_w = 0;
    if (std::abs(az_w) < 0.3) az_w = 0;
}
// Двойное интегрирование
n.vx += ax_w * dt; n.vy += ay_w * dt; n.vz += az_w * dt;
n.px += n.vx * dt; n.py += n.vy * dt; n.pz += n.vz * dt;
\end{lstlisting}

\subsection{Вычисление углов Эйлера}

Завершает функцию вычисление углов Эйлера по формулам
\eqref{eq:roll}--\eqref{eq:yaw} с переводом в градусы. Зажим
аргумента \texttt{asin} в интервал $[-1, 1]$ предотвращает исключения
при численном выходе за границу из-за накопленной погрешности:

\begin{lstlisting}[caption={Углы Эйлера из кватерниона (конвенция ZYX)}]
const double R2D = 180.0 / M_PI;
n.roll  = std::atan2(2*(qw*qx + qy*qz),
                     1-2*(qx*qx + qy*qy)) * R2D;
double sp = std::max(-1.0, std::min(1.0, 2*(qw*qy - qz*qx)));
n.pitch = std::asin(sp) * R2D;
n.yaw   = std::atan2(2*(qw*qz + qx*qy),
                     1-2*(qy*qy + qz*qz)) * R2D;
\end{lstlisting}

\subsection{Шейдеры OpenGL 3.3}

Визуализатор использует две пары GLSL-шейдеров. Первая
(\texttt{VERT3D} / \texttt{FRAG3D}) предназначена для рендеринга
3D-модели полётного контроллера с диффузным освещением по
модели Ламберта:

\begin{lstlisting}[language={},caption={Вершинный шейдер 3D-модели}]
#version 330 core
layout(location=0) in vec3 aPos;
layout(location=1) in vec3 aColor;
layout(location=2) in vec3 aNormal;
out vec3 vColor; out vec3 vNormal; out vec3 vPos;
uniform mat4 uMVP; uniform mat4 uModel;
void main(){
    vPos    = vec3(uModel * vec4(aPos,1));
    vNormal = mat3(transpose(inverse(uModel))) * aNormal;
    vColor  = aColor;
    gl_Position = uMVP * vec4(aPos,1);
}
\end{lstlisting}

\begin{lstlisting}[language={},caption={Фрагментный шейдер с освещением Ламберта}]
#version 330 core
in vec3 vColor; in vec3 vNormal; in vec3 vPos;
out vec4 outColor;
uniform vec3 uLight; uniform float uAlpha;
void main(){
    float d = max(dot(normalize(vNormal), normalize(uLight)), 0.0);
    outColor = vec4(vColor * (0.35 + 0.65*d), uAlpha);
}
\end{lstlisting}

Вторая пара (\texttt{VERT\_FLAT} / \texttt{FRAG\_FLAT}) используется
для отрисовки всех линейных примитивов без освещения: сетки, мировых
осей, траектории и HUD-панелей.

\subsection{Геометрия модели полётного контроллера}

3D-модель FC строится процедурно из параллелепипедов. Функция
\texttt{push\_box()} генерирует шесть граней прямоугольника с
различными оттенками для имитации объёма. Вспомогательная
\texttt{push\_arrow()} строит цилиндрические стрелки осей:

\begin{lstlisting}[caption={Сборка процедурной модели FC}]
static std::vector<V3> build_fc()
{
    std::vector<V3> v;
    // Корпус -- тёмно-серый прямоугольник
    push_box(v, 0,0,0,   0.8f,0.8f,0.14f, 0.13f,0.14f,0.19f);
    // Полоска "нос" (направление Y+)
    push_box(v, 0,0.37f,0.08f, 0.7f,0.06f,0.04f, 0.95f,0.95f,0.95f);
    // Рёбра для объёма
    push_box(v, 0.37f,0,0.08f, 0.04f,0.8f,0.04f, 0.22f,0.22f,0.28f);
    push_box(v,-0.37f,0,0.08f, 0.04f,0.8f,0.04f, 0.22f,0.22f,0.28f);
    // Оси: X=красный, Y=зелёный, Z=синий
    push_arrow(v,0,0,0.1f, 1,0,0, 0.7f, 0.95f,0.10f,0.10f);
    push_arrow(v,0,0,0.1f, 0,1,0, 0.7f, 0.10f,0.92f,0.10f);
    push_arrow(v,0,0,0.07f,0,0,1, 0.7f, 0.15f,0.42f,0.98f);
    return v;
}
\end{lstlisting}

\subsection{Кольцевой буфер для HUD-графиков}

Для отображения временных рядов в нижней панели используется структура
\texttt{RingBuf} на основе \texttt{std::deque}, хранящая последние
$N = 300$ значений с автоматическим вытеснением устаревших:

\begin{lstlisting}[caption={Кольцевой буфер для временных рядов}]
struct RingBuf {
    std::deque<float> d;
    int maxN;
    explicit RingBuf(int n=300): maxN(n){}
    void push(float v){
        d.push_back(v);
        if((int)d.size()>maxN) d.pop_front();
    }
};
\end{lstlisting}

Визуализатор поддерживает 12 независимых буферов: по три для
акселерометра, гироскопа, углов Эйлера и скорости.

\subsection{Главный цикл рендера}

Функция \texttt{run\_visualizer()} инициализирует контекст GLFW/GLEW,
компилирует шейдеры, загружает геометрию в видеопамять и запускает
главный цикл. Схема каждого кадра:
\begin{enumerate}[leftmargin=2cm]
    \item Обработать события ввода (мышь, клавиши).
    \item Вычитать из \texttt{SharedBuffer} \textit{все} накопившиеся
    пакеты, вызвать для каждого \texttt{integrate\_imu()}, добавить
    точку траектории и обновить кольцевые буферы.
    \item Обновить матрицы орбитальной камеры, следующей за FC.
    \item Отрисовать \textbf{3D-сцену} (верхние 72\% экрана):
    сетку, мировые оси, траекторию с затуханием цвета, модель FC.
    \item Отрисовать \textbf{2D HUD} (нижние 30\% экрана): четыре
    панели с графиками акселерометра, гироскопа, углов Эйлера

\end{enumerate}

\subsection{Результаты измерения ориентации}

В ходе тестирования подтверждена корректная работа алгоритма
кватернионного интегрирования. При вращении полётного контроллера
по всем трём осям трёхмерная модель воспроизводила движения с высокой
точностью. Углы Эйлера, вычисленные из кватерниона по
формулам~\eqref{eq:roll}--\eqref{eq:yaw}, соответствовали физическим
поворотам устройства. На HUD-панели гироскопа кривые угловых скоростей
чётко отображали начало и конец вращения, при покое сходясь к нулю
--- что подтверждает корректность подобранных смещений~\eqref{eq:biases}.

\subsection{Результаты оценки позиции}

Алгоритм двойного интегрирования ускорения показал
неудовлетворительные результаты в части оценки реальных координат.
Ключевая проблема~--- \textbf{дрейф позиции}: уже через 5--10~секунд
движения накопленная ошибка составляла метры, не соответствующие
реальному перемещению. Механизм ZUPT снизил скорость дрейфа в
состоянии покоя, однако не устранил его полностью из-за ненулевого
шума акселерометра порядка $10^{-3}$--$10^{-2}$~м/с$^2$.

\subsection{Анализ причин дрейфа}

Пусть истинное ускорение $a(t)$ и измеренное $\hat{a}(t)$ связаны:
\begin{equation}
    \hat{a}(t) = a(t) + b + \eta(t),
    \label{eq:noise}
\end{equation}
где $b$~--- постоянное смещение после калибровки, $\eta(t)$~---
случайный шум. Ошибка позиции после двойного интегрирования:
\begin{equation}
    \delta p(T) = \frac{1}{2}\,b\,T^2 + \int_0^T\!\!(T - t)\,\eta(t)\,dt.
    \label{eq:drift}
\end{equation}

Первое слагаемое растёт \textit{квадратично} по времени даже при
малом остаточном смещении~\cite{groves2013}. За 10~секунд это даёт
ошибку позиции порядка единиц метров~\cite{woodman2007}.

\subsection{Методы устранения дрейфа}

Полноценная инерциальная навигация невозможна без внешних корректирующих
сенсоров~\cite{groves2013}. Возможные направления развития системы:
\begin{enumerate}[leftmargin=2cm]
    \item \textbf{Фильтр Калмана с GPS-коррекцией}~\cite{groves2013}~---
    классический подход для авиационных INS. GPS обеспечивает
    абсолютную позицию с частотой 1--10~Гц; IMU~--- высокочастотное
    обновление ориентации на 100--1000~Гц.
    \item \textbf{Фильтр Маджвика}~\cite{madgwick2011}~---
    градиентный алгоритм слияния гироскопа, акселерометра и
    магнитометра. Существенно улучшает оценку ориентации, но не
    решает проблему позиции.
    \item \textbf{Нелинейный дополняющий фильтр на SO(3)}~\cite{mahony2008}~---
    альтернативный подход к оценке ориентации.
    \item \textbf{Оптический поток}~\cite{px4_docs}~---
    оценка скорости относительно поверхности по видеопотоку;
    применяется в системах без GPS.
    \item \textbf{ZUPT с фильтром Калмана}~\cite{skog2010}~---
    расширенное применение метода нулевой скорости в рамках
    оптимального оценщика состояния.
\end{enumerate}

Реализованы следующие компоненты:
\begin{itemize}[leftmargin=2cm]
    \item модуль приёма и парсинга сообщений \texttt{RAW\_IMU} протокола
    MAVLink~\cite{mavlink_docs} с записью данных в CSV;
    \item алгоритм калибровки сырых показаний с экспериментально
    определёнными смещениями нуля;
    \item точное кватернионное интегрирование угловой скорости через
    ось-угол~\cite{kuipers1999} с нормализацией на каждом шаге;
    \item вычисление матрицы вращения и углов Эйлера из кватерниона;
    \item алгоритм двойного интегрирования ускорения~\cite{groves2013};
    \item механизм ZUPT~\cite{skog2010} для частичной компенсации дрейфа;
    \item многопоточная архитектура C++14 с потокобезопасным буфером;
    \item OpenGL~3.3-визуализация с GLSL-шейдерами, процедурной
    3D-моделью FC, траекторией с цветовым затуханием и HUD-панелью
    с четырьмя графиками датчиков~\cite{glfw_docs}.
\end{itemize}

Алгоритм оценки ориентации показал высокое качество. Алгоритм оценки
позиции продемонстрировал принципиальную проблему накопления дрейфа,
характерную для МЭМС-датчиков~\cite{titterton2004,woodman2007}: ошибка
нарастает квадратично по времени. Получение пригодных для практики
координат возможно лишь при совместной обработке данных IMU с
корректирующими сенсорами в рамках фильтра Калмана или аналогичного
оптимального оценщика~\cite{groves2013}.
